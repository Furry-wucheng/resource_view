# Issues — 协作提案

> 新增需求、Bug 修复、性能优化的统一入口。
> 人填写描述和验收标准，AI 补充影响分析和执行计划。

---

## 命名规则

```
{YYYY-MM-DD}-{feature|bug|perf}-{简短描述}.md
```

示例：
- `2026-06-25-feature-章节画廊模式.md`
- `2026-06-25-bug-数据库迁移失败.md`
- `2026-06-25-perf-首页网格滚动卡顿.md`

## 状态流转

```
⬜ 待办 → 🟡 进行中 → ✅ 已完成 → ❌ 已关闭（不做了）
```

## 流程

1. **人** — 按下方对应模板新建文件，填写描述和验收标准
2. **AI（plan 阶段）** — 补充影响分析和执行计划，标记 🟡
3. **AI（执行阶段）** — 按 worktree + TDD 流程开发
4. **AI（完成）** — 填写产出清单，标记 ✅
5. **合并** — 若提案标注"需合并回原文档"，AI 在完成后更新对应的 share/prd/tech 文档

## 合并规则

提案标注"需合并回原文档"时，AI 完成后按以下规则更新：

| 目标文档类型 | 更新方式 |
|-------------|---------|
| `share/` 共享契约 | 不可直接修改。在 decisions/ 中记录变更建议，走 PR review |
| `prd/` 产品文档 | 直接在对应 PRD 文件中追加或修改 |
| `tech/` 技术文档 | 直接在对应技术文档中追加或修改 |
| `mvp/` 阶段文档 | 新增 stage 追加到进度表末尾（M29+），更新依赖图 |
| `decisions/` 决策日志 | 创建对应 decision 记录，更新 cross-reference |

## 排查清单

AI 进入 `doc/` 任何子目录前，必须先读该目录的 AGENTS.md，再进入具体文档。
AI 在 plan 阶段看到 `doc/issues/` 下有 ⬜ 待办的提案时，应主动接手分析。
